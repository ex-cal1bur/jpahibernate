# 1.4. JDBC a pelo vs ORM

## Objetivo de este apartado

En el apartado anterior hablamos en abstracto del *impedance mismatch* y de qué hace un ORM. Aquí lo bajamos a tierra: vamos a ver **el mismo caso de uso resuelto con JDBC puro y resuelto con JPA**, y a contar línea por línea qué te ahorra el ORM y qué decisiones nuevas te impone.

Importante: este apartado **no es para enseñar JDBC**. Es para que te duela JDBC. Cuando duele, se entiende por qué existió Hibernate.

---

## El caso de uso

Algo simple, sin relaciones todavía. Un usuario con id, nombre y email. Tres operaciones:

1. **Insertar** un nuevo usuario.
2. **Recuperarlo** por su id.
3. **Listar** todos los usuarios.

Mismo dominio, mismo objetivo, dos implementaciones.

---

## Versión 1: JDBC a pelo

Esto es lo que se escribía antes de los ORMs. Y lo que aún se escribe en proyectos donde por algún motivo no se usa uno.

### El POJO

```java
public class User {
    private Long id;
    private String name;
    private String email;

    public User() {}
    public User(String name, String email) {
        this.name = name;
        this.email = email;
    }

    // getters y setters
    public Long getId() { return id; }
    public void setId(Long id) { this.id = id; }
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
    public String getEmail() { return email; }
    public void setEmail(String email) { this.email = email; }
}
```

Hasta aquí inocente. El POJO es POJO.

### El DAO con JDBC

```java
public class UserDao {

    private static final String URL = "jdbc:h2:mem:demo";
    private static final String USER = "sa";
    private static final String PASS = "";

    public Long insert(User user) {
        String sql = "INSERT INTO users (name, email) VALUES (?, ?)";

        try (Connection conn = DriverManager.getConnection(URL, USER, PASS);
             PreparedStatement stmt = conn.prepareStatement(
                     sql, Statement.RETURN_GENERATED_KEYS)) {

            stmt.setString(1, user.getName());
            stmt.setString(2, user.getEmail());

            int affected = stmt.executeUpdate();
            if (affected == 0) {
                throw new RuntimeException("Insert falló, ninguna fila afectada");
            }

            try (ResultSet keys = stmt.getGeneratedKeys()) {
                if (keys.next()) {
                    Long generatedId = keys.getLong(1);
                    user.setId(generatedId);
                    return generatedId;
                } else {
                    throw new RuntimeException("Insert falló, no se obtuvo ID");
                }
            }
        } catch (SQLException e) {
            throw new RuntimeException("Error insertando usuario", e);
        }
    }

    public User findById(Long id) {
        String sql = "SELECT id, name, email FROM users WHERE id = ?";

        try (Connection conn = DriverManager.getConnection(URL, USER, PASS);
             PreparedStatement stmt = conn.prepareStatement(sql)) {

            stmt.setLong(1, id);

            try (ResultSet rs = stmt.executeQuery()) {
                if (rs.next()) {
                    User u = new User();
                    u.setId(rs.getLong("id"));
                    u.setName(rs.getString("name"));
                    u.setEmail(rs.getString("email"));
                    return u;
                }
                return null;
            }
        } catch (SQLException e) {
            throw new RuntimeException("Error consultando usuario", e);
        }
    }

    public List<User> findAll() {
        String sql = "SELECT id, name, email FROM users";
        List<User> result = new ArrayList<>();

        try (Connection conn = DriverManager.getConnection(URL, USER, PASS);
             PreparedStatement stmt = conn.prepareStatement(sql);
             ResultSet rs = stmt.executeQuery()) {

            while (rs.next()) {
                User u = new User();
                u.setId(rs.getLong("id"));
                u.setName(rs.getString("name"));
                u.setEmail(rs.getString("email"));
                result.add(u);
            }
            return result;

        } catch (SQLException e) {
            throw new RuntimeException("Error listando usuarios", e);
        }
    }
}
```

### El DDL que tienes que crear a mano

```sql
CREATE TABLE users (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    email VARCHAR(255)
);
```

Si cambias la clase Java, **tienes que cambiar la tabla a mano**. Si cambias la tabla, tienes que cambiar la clase, las queries, los binds y los mapeos. **Cada lado vive su vida**, y la sincronización es responsabilidad del programador.

---

## Lo que estás haciendo a mano en JDBC

Para tres operaciones básicas sobre una sola entidad, has tenido que:

- **Gestionar conexiones** abriendo y cerrando con try-with-resources (y rezar para que no falte nada).
- **Escribir SQL** literal, sin verificación en tiempo de compilación.
- **Parametrizar las queries** una a una con `setLong`, `setString` (y acordarte del orden, los índices empiezan en 1, no en 0).
- **Recoger las claves generadas** con `RETURN_GENERATED_KEYS` y un segundo `ResultSet`.
- **Mapear filas a objetos** columna por columna con sus tipos correctos.
- **Manejar `SQLException`** que es checked: o capturas o la propagas, no hay opción.
- **Cubrir todos los `else`**: ¿y si no devuelve filas? ¿y si no hay ID generado? ¿y si la conexión cae a la mitad?
- **Evitar SQL injection** acordándote siempre de usar `PreparedStatement` y nunca concatenar strings.

Y eso para una sola tabla. Imagina las cuatro entidades del modelo de pizzas con sus relaciones N:M y 1:N. Estimación: **600-800 líneas** de código DAO repetitivo, frágil ante cambios.

## Versión 2: el mismo caso con JPA + Hibernate

### La entidad

```java
import jakarta.persistence.*;

@Entity
@Table(name = "users")
public class User {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, length = 100)
    private String name;

    @Column(length = 255)
    private String email;

    public User() {}
    public User(String name, String email) {
        this.name = name;
        this.email = email;
    }

    // getters y setters (los mismos)
}
```

Lo que ha cambiado:

- Cinco anotaciones: `@Entity`, `@Table`, `@Id`, `@GeneratedValue`, `@Column`.
- **Cero SQL.**
- **Cero código de mapeo.**

### El "DAO" con JPA

```java
import jakarta.persistence.*;
import java.util.List;

public class UserService {

    private final EntityManagerFactory emf;

    public UserService(EntityManagerFactory emf) {
        this.emf = emf;
    }

    public Long insert(User user) {
        EntityManager em = emf.createEntityManager();
        try {
            em.getTransaction().begin();
            em.persist(user);
            em.getTransaction().commit();
            return user.getId();
        } finally {
            em.close();
        }
    }

    public User findById(Long id) {
        EntityManager em = emf.createEntityManager();
        try {
            return em.find(User.class, id);
        } finally {
            em.close();
        }
    }

    public List<User> findAll() {
        EntityManager em = emf.createEntityManager();
        try {
            return em.createQuery("SELECT u FROM User u", User.class)
                     .getResultList();
        } finally {
            em.close();
        }
    }
}
```

### El DDL... no lo escribes

Hibernate puede generarlo a partir de las anotaciones de la entidad. Con la propiedad `hibernate.hbm2ddl.auto=create-drop`, cuando arranca la aplicación se crean las tablas a partir de las clases. Cuando cierra, se borran.

En producción esto **no se hace** (es peligroso): se usan migraciones controladas con Flyway o Liquibase. Pero para desarrollo y formación es perfecto.

---

## La diferencia, lado a lado

| Aspecto | JDBC | JPA |
|---|---|---|
| Líneas para 3 operaciones | ~80 | ~25 |
| SQL escrito a mano | Sí, todo | Casi nada (JPQL puntual) |
| Mapeo de filas a objetos | Manual columna a columna | Automático por anotaciones |
| Manejo de conexiones | Manual con try-with-resources | Encapsulado en `EntityManager` |
| `SQLException` checked | Sí, ineludible | Hibernate las traduce a runtime |
| Generación de IDs | Manual con `getGeneratedKeys` | Automática con `@GeneratedValue` |
| DDL | Escribes y mantienes | Hibernate puede generarlo |
| Cambio de motor de BD | Reescribir SQL específico | Cambias dialecto y URL |
| Cache de objetos | Implementas tú | Incluida en el `EntityManager` |
| Detección de cambios | Implementas tú | *Dirty checking* automático |
| Verificación en compilación | El SQL no se valida | JPQL se puede validar parcialmente |

---

## Lo que JPA te impone a cambio

No es gratis. Lo que ganas en automatización lo pagas con un modelo mental nuevo:

1. **Tienes que conocer el ciclo de vida de las entidades.** *Transient*, *managed*, *detached*, *removed*. Si no lo entiendes, los cambios no se guardan o se guardan dos veces.
2. **Tienes que entender el contexto de persistencia.** Qué es, cuánto vive, cómo se relaciona con la transacción.
3. **Tienes que decidir cuándo cargar las relaciones.** Lazy o eager. Mal decidido, problema de rendimiento garantizado.
4. **El SQL generado no lo controlas tú directamente.** Cuando es ineficiente, tienes que entender qué genera Hibernate y cómo guiarlo.
5. **Las anotaciones tienen comportamientos sutiles.** `cascade`, `orphanRemoval`, `mappedBy`... cada una con consecuencias que no son obvias.

A cambio, dejas de ser un copista de SQL y te dedicas a **diseñar tu modelo de dominio**.

---

## Una observación honesta

JDBC no es "malo" ni "viejo". Sigue teniendo casos de uso legítimos:

- **Queries muy específicas** donde necesitas control absoluto del SQL.
- **Queries de altísimo rendimiento** donde el ORM mete overhead inaceptable.
- **Reportes complejos** que ningún ORM mapea bien.
- **Stored procedures** y operaciones que viven en el motor de BD.

Lo que ha quedado obsoleto **no es JDBC**, sino **escribir todo el código persistente con JDBC**. Hoy es razonable un híbrido: ORM para el 95% de operaciones CRUD, y JDBC (o JPQL nativo) para casos puntuales.

De hecho, JPA **incluye** una vía para usar SQL nativo cuando lo necesitas: `entityManager.createNativeQuery(...)`. Lo veremos al final del curso.

---

## El cambio de mentalidad

Si vienes de JDBC, JPA al principio es desconcertante. Eres tú quien escribe el SQL en JDBC; en JPA, alguien lo escribe por ti, y ese alguien tiene que tener acceso a metadatos, sesiones, contextos, transacciones. Hay más piezas en juego.

Y hay una idea que cuesta interiorizar: **en JPA no insertas filas, persistes objetos**. No actualizas columnas, modificas atributos y al hacer commit los cambios viajan. No borras filas, llamas a `remove(entidad)`.

El modelo mental cambia. El SQL sigue ahí debajo, pero tú ya no lo escribes la mayor parte del tiempo: lo verificas (con `show_sql`), lo guías (con anotaciones y JPQL) y, en último recurso, lo bypaseas (con SQL nativo).

---

## Resumen

- **JDBC** te da control total y te obliga a escribir todo el plumbing a mano.
- **JPA** automatiza la traducción objeto ↔ tabla a cambio de imponer un modelo mental.
- El balance favorece a JPA en el 95% de las operaciones CRUD de aplicaciones de gestión.
- JDBC sigue teniendo nicho: queries críticas, reportes, stored procedures.
- El cambio mental clave: **dejas de manipular filas y empiezas a manipular objetos**.
- A cambio, hay que aprender el contexto de persistencia, el ciclo de vida de las entidades y las decisiones de carga.

El resto del curso es, esencialmente, **aprender a manejar lo que JPA te impone a cambio de lo que te ahorra**.
