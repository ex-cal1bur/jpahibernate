# 1.5. JPA, Hibernate y Jakarta EE: especificación, implementación, paraguas

## Por qué este apartado

Tres palabras que el principiante mezcla constantemente y que no son lo mismo:

- **JPA**
- **Hibernate**
- **Jakarta EE**

Si trabajas con cualquier proyecto Java empresarial en 2026, tienes que tener clarísima la diferencia. Mezclar estos términos es síntoma de no entender la arquitectura del ecosistema.

---

## La distinción fundamental: especificación vs implementación

Empecemos por el concepto más importante. En el mundo Java empresarial, hay una separación deliberada entre:

### Una **especificación**

Un documento (literalmente, un PDF de cientos de páginas) que define **qué API debe existir, cómo se llama cada clase, qué hace cada método, qué reglas debe cumplir el comportamiento**.

La especificación es **un contrato escrito**, no código ejecutable. Define:

- Las interfaces y clases públicas.
- Los métodos y sus firmas.
- Las anotaciones disponibles.
- El comportamiento esperado en cada caso.
- Las excepciones que pueden lanzarse.

### Una **implementación**

Una librería de código real, ejecutable, que **cumple lo que la especificación pide**. Compilable. Importable como dependencia. Funcional.

Pueden existir **múltiples implementaciones** de una misma especificación. Cada una con sus propias decisiones internas, optimizaciones y extensiones, pero todas obligadas a respetar el contrato común.

### El beneficio de la separación

Esta separación es la razón de que en Java puedas cambiar de implementación sin cambiar tu código:

> Si tu código solo depende de la especificación (de las interfaces y clases del API estándar), puedes sustituir la implementación por otra y, en teoría, todo sigue funcionando.

En la práctica, hay matices: cada implementación tiene extensiones propias, características específicas, configuración distinta. Pero el código de negocio que use solo el API estándar es portable.

---

## JPA — La especificación

**JPA = Java Persistence API.**

Es **la especificación**. Un documento mantenido por la Eclipse Foundation (antes Oracle, antes Sun) que define cómo debe comportarse cualquier librería que diga ser "compatible con JPA".

JPA define:

- Las anotaciones: `@Entity`, `@Id`, `@Column`, `@OneToMany`, `@ManyToOne`...
- Las interfaces principales: `EntityManager`, `EntityManagerFactory`, `EntityTransaction`, `Query`, `TypedQuery`, `CriteriaBuilder`...
- El lenguaje JPQL.
- El ciclo de vida de las entidades.
- Las reglas de transacciones, contextos de persistencia, estados.

**Lo que JPA NO es:**

- No es código ejecutable. La especificación viene acompañada del *API JAR* (las interfaces sin implementación), pero ese JAR por sí solo no persiste nada.
- No es un framework. Es solo un contrato.
- No es propiedad de una empresa privada. Es estándar abierto bajo Eclipse Foundation.

### Versiones de JPA

| Versión JPA | Año | Paquete | Notas |
|---|---|---|---|
| JPA 1.0 | 2006 | `javax.persistence` | Inicial. |
| JPA 2.0 | 2009 | `javax.persistence` | Criteria API, listas ordenadas. |
| JPA 2.1 | 2013 | `javax.persistence` | Stored procedures, entity graphs. |
| JPA 2.2 | 2017 | `javax.persistence` | Soporte de tipos `java.time`. |
| **Jakarta Persistence 3.0** | 2020 | **`jakarta.persistence`** | Rebranding tras donación a Eclipse. |
| Jakarta Persistence 3.1 | 2022 | `jakarta.persistence` | UUID nativo, funciones numéricas. |
| Jakarta Persistence 3.2 | 2024 | `jakarta.persistence` | Mejoras menores. |

A partir de la versión 3.0, "JPA" pasa oficialmente a llamarse **"Jakarta Persistence"**. La gente sigue diciendo "JPA" coloquialmente, pero el nombre oficial cambió.

## Hibernate — Una implementación

**Hibernate ORM** es **la implementación más popular** de JPA. Hay otras (las veremos), pero Hibernate domina el mercado con una cuota cercana al 90% en proyectos Java empresariales.

Hibernate implementa todo lo que JPA pide y, además, **añade extensiones propias**. Cosas que JPA no contempla pero Hibernate sí:

- **Anotaciones específicas** como `@Type`, `@Formula`, `@Filter`, `@Where`, `@SQLInsert`.
- **Tipos personalizados** para mapear cosas que JPA no contempla nativamente.
- **Caché de segundo nivel** con providers (Ehcache, Infinispan...).
- **Estadísticas detalladas** de queries, cache hits, etc.
- **HQL** (Hibernate Query Language), antecesor de JPQL y un poco más expresivo.
- **Hibernate Search** (para integración con Lucene/Elasticsearch).
- **Hibernate Envers** (para auditoría de cambios).
- **Hibernate Validator** (que de hecho es la implementación oficial de Bean Validation).

### El precio de las extensiones

Si en tu proyecto usas solo lo que JPA define, eres "portable": podrías cambiar Hibernate por otra implementación con cambios mínimos.

Si usas extensiones específicas de Hibernate (`@Formula`, `@Filter`, etc.), **te casas con Hibernate**. No es necesariamente malo (Hibernate es robusto, vivo, ampliamente soportado), pero hay que ser consciente de la decisión.

### Historia breve

- **2001**: Gavin King crea Hibernate como respuesta al desastroso EJB 2.0.
- **2006**: Hibernate inspira directamente la primera versión de JPA.
- **2010**: Hibernate se convierte en la implementación de referencia de JPA en JBoss/WildFly.
- **2022**: Hibernate 6.0 migra a `jakarta.persistence` cumpliendo Jakarta Persistence 3.0.

Es la única librería de Java empresarial que **inspiró el estándar que después implementaría**. Eso la hace particularmente coherente con JPA.

---

## Otras implementaciones de JPA

Aunque Hibernate domina, **no es la única opción**. Saberlo es importante:

| Implementación | Mantenedor | Notas |
|---|---|---|
| **Hibernate ORM** | Red Hat | Líder de mercado. Default de Spring Boot. |
| **EclipseLink** | Eclipse Foundation | Implementación de referencia oficial de Jakarta Persistence. |
| **OpenJPA** | Apache | Mantenida pero con poca actividad reciente. |
| **DataNucleus** | DataNucleus | Soporta JPA y JDO, pensada para entornos no relacionales. |

En la práctica, en cualquier empresa que vayas a trabajar, será **Hibernate al 95%**. EclipseLink aparece a veces en proyectos GlassFish/Payara. Las otras dos son raras.

Lo importante es entender la arquitectura: **JPA es el API; Hibernate es una opción**.

---

## Jakarta EE — El paraguas

Aquí entra el tercer término, que es el que más confunde porque es el más amplio.

**Jakarta EE = Jakarta Enterprise Edition.**

Es una **plataforma de especificaciones para aplicaciones Java empresariales**. Un conjunto de APIs estandarizadas para hacer:

- Persistencia → **Jakarta Persistence** (lo que llamamos JPA).
- Servicios web REST → **Jakarta REST** (lo que se llamaba JAX-RS).
- Servicios web SOAP → Jakarta XML Web Services (JAX-WS).
- Inyección de dependencias → **Jakarta CDI** (Contexts and Dependency Injection).
- Validación → **Jakarta Validation** (Bean Validation).
- Mensajería → **Jakarta Messaging** (JMS).
- Servlets → **Jakarta Servlet**.
- Páginas web → Jakarta Server Faces (JSF), Jakarta Server Pages (JSP).
- Transacciones → **Jakarta Transactions** (JTA).
- Seguridad → Jakarta Security.
- Y más.

**JPA es UNA de las muchas especificaciones** que componen Jakarta EE.

### La relación entre los tres conceptos

```
Jakarta EE  (paraguas)
    ├── Jakarta Persistence (JPA)
    │     └── Implementación: Hibernate, EclipseLink, ...
    ├── Jakarta REST
    │     └── Implementación: Jersey, RESTEasy, ...
    ├── Jakarta CDI
    │     └── Implementación: Weld, OpenWebBeans, ...
    └── ...
```

### Java EE → Jakarta EE

El cambio de nombre es el mismo que vimos antes: en 2017 Oracle dona Java EE a Eclipse, y desde 2020 todo el ecosistema se llama Jakarta EE con paquetes `jakarta.*`.

### Servidores de aplicaciones que implementan Jakarta EE

Históricamente, las aplicaciones Java empresariales se desplegaban en "servidores de aplicaciones" que implementaban toda la pila Jakarta EE:

- **WildFly** (antes JBoss).
- **Payara** (basado en GlassFish).
- **Open Liberty** (de IBM).
- **TomEE**.

Hoy en día, con la llegada de Spring Boot y Quarkus, este modelo de "servidor monolítico" ha perdido protagonismo. Pero sigue muy vivo en banca, seguros, administración pública.

## Spring vs Jakarta EE: el otro mundo

Hasta aquí hemos hablado del mundo Jakarta EE. Pero existe **otro ecosistema paralelo**: Spring.

- **Jakarta EE** → especificaciones estandarizadas con varias implementaciones.
- **Spring** → un framework que **usa algunas especificaciones de Jakarta EE** (como JPA, Bean Validation) pero **no implementa la pila entera**, y añade sus propias soluciones para el resto (Spring MVC en lugar de JAX-RS, Spring DI en lugar de CDI, etc.).

**¿Cómo encaja JPA + Hibernate en un proyecto Spring?**

Spring no implementa JPA. Lo que hace Spring (concretamente Spring Data JPA) es:

- Usar **una implementación de JPA** (por defecto Hibernate) como motor de persistencia.
- Añadir azúcar sintáctico encima: `@Repository`, `JpaRepository`, queries derivadas...
- Encargarse de la configuración, el `EntityManager` y las transacciones por debajo.

Por eso el código de un proyecto Spring usa anotaciones JPA estándar (`@Entity`, `@Id`...) pero también `@Repository`, `@Transactional`, etc. Las primeras son JPA. Las segundas son Spring.

En **este curso no usamos Spring**. Trabajamos con JPA puro + Hibernate, viendo qué hay debajo. Pero hay que tener clara la relación, porque cuando un alumno salga al mercado se encontrará casi siempre con Spring por delante.

## Aclaración con los nombres confusos

Repasemos los términos que hemos visto y que la gente confunde:

| Término | Qué es |
|---|---|
| **Java SE** | El lenguaje Java y su API básica (no empresarial). |
| **Java EE** | Plataforma empresarial vieja (hasta 2017, propiedad de Oracle). |
| **Jakarta EE** | Plataforma empresarial nueva (desde 2017, en Eclipse). |
| **JPA** | Especificación de persistencia. Una de las APIs de Jakarta EE. |
| **Jakarta Persistence** | Nuevo nombre oficial de JPA desde 2020. |
| **Hibernate** | Implementación concreta de JPA. La más usada. |
| **Spring** | Framework alternativo que usa partes de Jakarta EE. |
| **Spring Data JPA** | Capa Spring sobre JPA. Suele usar Hibernate por debajo. |

Cuando un programador novato dice "voy a usar Hibernate":

- Si está en un servidor Jakarta EE → probablemente JPA + Hibernate dentro de Wildfly/Payara.
- Si está en Spring Boot → probablemente Spring Data JPA + Hibernate.
- Si está en un proyecto pelado → JPA + Hibernate directo, como en este curso.

Las tres opciones son válidas. Y todas usan Hibernate por debajo. La diferencia es la capa que tienes encima.

---

## Lo que esto te permite hacer en una entrevista

Si te preguntan "¿qué relación hay entre JPA, Hibernate y Jakarta EE?", una buena respuesta es:

> "JPA es una especificación de persistencia que forma parte de Jakarta EE, la plataforma de APIs estándar para aplicaciones Java empresariales. Hibernate es la implementación más popular de esa especificación. Si yo escribo código contra el API de JPA, en teoría podría cambiar Hibernate por EclipseLink u OpenJPA con cambios mínimos. En la práctica, la mayoría de proyectos usan Hibernate porque es maduro, está bien soportado y es el default de Spring Boot."

Esa respuesta separa al que ha entendido la arquitectura del que se ha aprendido las anotaciones.

---

## Resumen

- **Java EE** era la plataforma empresarial vieja, propiedad de Oracle. Murió en 2017.
- **Jakarta EE** es la plataforma empresarial actual, mantenida por la Eclipse Foundation.
- **JPA** (Jakarta Persistence) es **una especificación** de persistencia. **Una de las muchas APIs** de Jakarta EE.
- **Hibernate** es **una implementación** de JPA. La más usada del mercado.
- Existen otras implementaciones (EclipseLink, OpenJPA, DataNucleus), pero son minoritarias.
- **Spring no es Jakarta EE**: es un framework alternativo que usa algunas de sus especificaciones (JPA, Bean Validation) y reemplaza otras con soluciones propias.
- **Programar contra el API estándar (JPA) te da portabilidad teórica**. Usar extensiones de Hibernate te ata a Hibernate.
- En la práctica profesional: 95% del tiempo, Hibernate. Pero entender la separación es lo que distingue al programador con criterio.
