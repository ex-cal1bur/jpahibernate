# 1.2. Comparativa de versiones: Java, Hibernate, `javax` vs `jakarta`

## Por qué este apartado existe

En el apartado anterior vimos que las versiones importan. Ahora vamos a desmenuzarlo: **qué versiones existen, qué pidieron en su momento, cómo se han ido encadenando, y qué decidir hoy en 2026**.

Este apartado tiene un objetivo muy concreto: que cuando salgas del curso y abras un proyecto en una empresa, sepas **leer el `pom.xml`** y entender en qué año tecnológico está atrapado ese proyecto.

---

## La línea temporal de Java

| Versión | Año | Tipo | Notas relevantes |
|---|---|---|---|
| Java 8 | 2014 | LTS | Lambdas, Streams. Aún muy usada en banca y legacy. |
| Java 11 | 2018 | LTS | Primera LTS post-modular. Var locals, HTTP Client. |
| Java 17 | 2021 | LTS | Records, sealed classes, pattern matching. |
| Java 21 | 2023 | LTS | Virtual threads, pattern matching para `switch`. |
| Java 25 | 2025 | LTS | Última LTS publicada. |

**Las versiones LTS (Long-Term Support)** son las que las empresas usan. Las intermedias (12, 13, 14, 18, 19, 20...) existen pero apenas se adoptan en producción porque pierden soporte rápido.

**Lo que esto significa para JPA:** la mayoría de proyectos reales en 2026 están en Java 17 o Java 21. Java 8 está desapareciendo (pero no del todo). Java 11 está en transición. Por encima de 21 es minoritario salvo en empresas muy actualizadas.

---

## La línea temporal de Hibernate

| Versión | Año | Java mínimo | API |
|---|---|---|---|
| Hibernate 4.x | 2012 | Java 6 | `javax.persistence` (JPA 2.0) |
| Hibernate 5.0 – 5.2 | 2015 | Java 8 | `javax.persistence` (JPA 2.1) |
| Hibernate 5.3 – 5.6 | 2018 | Java 8 | `javax.persistence` (JPA 2.2) |
| **Hibernate 6.0** | 2022 | Java 11 | **`jakarta.persistence`** (JPA 3.0) |
| Hibernate 6.1 | 2022 | Java 11 | `jakarta.persistence` |
| Hibernate 6.2 | 2023 | **Java 17** | `jakarta.persistence` |
| Hibernate 6.4 – 6.6 | 2023 – 2024 | Java 17 | `jakarta.persistence` |
| Hibernate 7.0 | 2025 | Java 17 | `jakarta.persistence` (JPA 3.2) |

**El salto crítico es 5.x → 6.x.** No es solo una actualización: cambia el paquete base (`javax` → `jakarta`), y por tanto **todas las clases de la aplicación que usen JPA deben tocar sus imports**. No es retrocompatible. Es una migración real, no un upgrade.

**Hibernate 6.2 es el segundo salto crítico.** A partir de ahí, Java 11 deja de funcionar. Si tu empresa está en Java 11 y quiere ir a Hibernate 7, primero tiene que actualizar Java.

---

## El cisma `javax` → `jakarta`

Esta es la historia que más confunde a quien empieza. Vamos a contarla bien para que no quede ninguna duda.

### La situación previa (hasta 2017)

- Existía algo llamado **Java EE** (Java Enterprise Edition).
- Era propiedad de **Oracle**.
- Sus paquetes vivían bajo `javax.*`: `javax.persistence`, `javax.servlet`, `javax.ejb`...
- JPA era una de sus especificaciones.

### El traspaso (2017)

- Oracle decide que Java EE no es estratégico y lo dona a la **Eclipse Foundation**.
- Eclipse lo renombra a **Jakarta EE**.
- Pero Oracle se reserva el derecho sobre el namespace `javax.*`. Eclipse no puede seguir publicando bajo ese paquete.

### La transición (2017 – 2020)

- Jakarta EE 8 (septiembre 2019): última versión que aún usa `javax.*`. Funcionalmente igual a Java EE 8.
- Jakarta EE 9 (noviembre 2020): **rebranding completo a `jakarta.*`**. Cero cambios funcionales, solo el nombre de los paquetes.

### El presente

- **Java EE → muerto.** No hay más versiones bajo ese nombre.
- **Jakarta EE → vivo.** Versiones 9, 10, 11, 12. Todas con `jakarta.*`.
- **`javax.persistence` → fósil.** Aún funciona si usas Hibernate 5.x, pero no recibe novedades. Cualquier librería moderna ha migrado.

### Cómo afecta esto a un `pom.xml`

Para entender qué versión usa un proyecto, mira sus dependencias:

```xml
<!-- Proyecto VIEJO (mundo javax) -->
<dependency>
    <groupId>org.hibernate</groupId>
    <artifactId>hibernate-core</artifactId>
    <version>5.6.15.Final</version>
</dependency>
```

```xml
<!-- Proyecto MODERNO (mundo jakarta) -->
<dependency>
    <groupId>org.hibernate.orm</groupId>
    <artifactId>hibernate-core</artifactId>
    <version>6.4.4.Final</version>
</dependency>
```

Fíjate en dos detalles:

1. **`groupId` cambia**: `org.hibernate` → `org.hibernate.orm`.
2. **`version` cambia de familia**: 5.x → 6.x.

Y en el código:

```java
// Mundo javax — código viejo, no compila contra Hibernate 6
import javax.persistence.Entity;
import javax.persistence.Id;

// Mundo jakarta — código moderno
import jakarta.persistence.Entity;
import jakarta.persistence.Id;
```

**Las clases tienen el mismo nombre.** Solo cambia el paquete. Esto hace que migrar un proyecto sea, en el caso simple, una sustitución masiva de imports. En el caso complejo, también hay que actualizar dependencias transitivas, drivers de servidor, librerías de terceros, etc.

---

## Por qué importa todo esto en clase

Cuando un alumno os abra el primer ejemplo del curso y vea:

```java
import jakarta.persistence.Entity;
```

Tiene que tener clarísimo:

- **Por qué `jakarta` y no `javax`.**
- **Que si copia código de un blog de hace cinco años, va a fallar al compilar.**
- **Que la solución no es retroceder a Hibernate 5, es entender que `javax` es legacy.**

Y cuando entréis en una empresa con un proyecto en Hibernate 5 + Java 8, vais a saber:

- Que estáis en un stack de 2018.
- Que la migración a Hibernate 6 es viable pero no trivial.
- Que requiere actualizar Java primero, luego Hibernate, luego limpiar imports.

## Tabla de decisión rápida

Si tienes que elegir versiones para un proyecto **nuevo en 2026**:

| Componente | Recomendación | Alternativa válida |
|---|---|---|
| Java | 21 LTS | 17 LTS |
| Hibernate | 6.6.x | 7.0.x |
| API JPA | `jakarta.persistence` 3.2 | (no hay alternativa moderna) |
| Build | Maven o Gradle | (cualquiera) |

Si te encuentras un proyecto **existente** en una empresa, lo primero que miras es:

1. ¿Java 8, 11, 17, 21? → fija el techo de Hibernate posible.
2. ¿`hibernate-core` 5.x o 6.x? → fija si estás en `javax` o `jakarta`.
3. ¿Imports en el código son `javax.persistence` o `jakarta.persistence`? → confirma.

Con esos tres datos sabes en qué año tecnológico vive el proyecto y qué tipo de migración tendría sentido.

## Nuestro stack en este curso

Para que conste en el material, lo que vamos a usar:

- **Java 17 o superior.**
- **Hibernate 6.4.4.Final.**
- **`jakarta.persistence` 3.x** (viene como dependencia transitiva de Hibernate).
- **H2** como base de datos.
- **Maven** como sistema de build.

Esto representa el estándar moderno en 2026. Salvo que tu empresa esté atrapada en versiones antiguas (cosa frecuente y legítima), este es el stack al que tu código debería tender.

---

## Resumen

- Java tiene versiones LTS cada 2-3 años. Las que importan en producción.
- Hibernate evoluciona ligado a Java: 5.x con Java 8/11 y `javax`, 6.x con Java 11/17 y `jakarta`, 7.x con Java 17 y `jakarta`.
- `javax.persistence` y `jakarta.persistence` son **el mismo API con paquetes distintos**. La migración es masiva pero conceptualmente trivial.
- Java EE (Oracle) → muerto. Jakarta EE (Eclipse) → vivo.
- Antes de copiar código de cualquier sitio, mira los imports para saber en qué mundo estás.
- Antes de elegir versiones para un proyecto nuevo, fija primero la versión de Java; el resto se deduce.
