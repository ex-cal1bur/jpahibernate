# 1.1. Caso real: el choque con el entorno

## Por qué empezamos por aquí

La mayoría de cursos arrancan con la teoría: qué es JPA, qué es un ORM, las cinco virtudes de Hibernate. Eso ya está en la documentación oficial y en mil tutoriales. Aquí vamos a empezar por algo distinto y mucho más útil: **qué te encuentras de verdad cuando intentas montar un proyecto Hibernate moderno en una máquina real**.

Esta es una historia auténtica de la preparación de este mismo curso. La cuento porque condensa, en cinco minutos, todo lo que vais a sufrir si os ponéis a trabajar con JPA y Hibernate sin contexto.

---

## La situación de partida

Punto de partida: una máquina Windows con un proyecto Maven recién creado en VS Code. Un `pom.xml` mínimo generado por el arquetipo por defecto:

```xml
<properties>
    <maven.compiler.source>11</maven.compiler.source>
    <maven.compiler.target>11</maven.compiler.target>
</properties>

<dependencies>
    <dependency>
        <groupId>junit</groupId>
        <artifactId>junit</artifactId>
        <version>4.11</version>
        <scope>test</scope>
    </dependency>
</dependencies>
```

Y en consola:

```
$ java --version
openjdk 11.0.16.1 2022-08-12 LTS
```

Java 11. JUnit 4.11. Sin Hibernate todavía. Todo aparentemente normal. Aparentemente.

---

## El primer problema: las versiones de Java

Lo primero que necesitamos es añadir Hibernate al proyecto. Hibernate Core, en su versión moderna, vive en el grupo `org.hibernate.orm`. Pero **¿qué versión cogemos?**

Aquí aparece la primera trampa. La tabla de compatibilidad real es:

| Versión Hibernate | Java mínimo | API que usa |
|---|---|---|
| Hibernate 5.6.x | Java 8 | `javax.persistence` |
| Hibernate 6.0 – 6.1 | Java 11 | `jakarta.persistence` |
| Hibernate 6.2 – 6.6 | **Java 17** | `jakarta.persistence` |
| Hibernate 7.x | **Java 17** | `jakarta.persistence` |

Traducido: con Java 11 te quedas atrapado en Hibernate 6.1 como mucho, una versión que ya no recibe novedades. Si quieres Hibernate moderno, necesitas Java 17 o superior.

Esto **no aparece en la página de inicio de Hibernate**. No es información destacada. Tienes que buscarla en las release notes, una a una.

---

## El segundo problema: `javax` vs `jakarta`

Si miras tutoriales de hace cinco años, todo el código JPA empieza así:

```java
import javax.persistence.Entity;
import javax.persistence.Id;
```

Si miras documentación moderna, empieza así:

```java
import jakarta.persistence.Entity;
import jakarta.persistence.Id;
```

¿Qué ha pasado? Una historia corporativa con consecuencias técnicas:

- Java EE pertenecía a Oracle. El paquete `javax.*` era propiedad de Oracle.
- En 2017 Oracle donó Java EE a la Eclipse Foundation, que lo renombró a **Jakarta EE**.
- Oracle no permitió a Eclipse seguir usando el paquete `javax.*`.
- A partir de Jakarta EE 9 (2020), todos los paquetes pasan a llamarse `jakarta.*`.

Resultado: **`javax.persistence` está muerto**. No recibe actualizaciones. Hibernate moderno no lo soporta. Si copias código de un tutorial viejo, no compila contra Hibernate 6.

¿Por qué esto importa para vosotros? Porque vais a encontrar:

- Material de formación con `javax.*` que no compila.
- Stack Overflow con respuestas que mezclan los dos.
- Proyectos legacy en empresas que aún usan `javax.*` y otros nuevos con `jakarta.*`.
- Migraciones de proyectos enteros consistentes solo en cambiar imports masivamente.

**Saber distinguir cuál usar y por qué no es opcional.**

## El tercer problema: el `JAVA_HOME` mentiroso

Volvemos a la máquina de la historia. Ejecutamos:

```
$ java --version
openjdk 11.0.16.1
```

Java 11. Pero también ejecutamos:

```
$ echo $JAVA_HOME
C:\Program Files\Java\jdk-18
```

¿Java 11 o Java 18? **Las dos cosas a la vez**, y eso es lo peligroso.

En Windows (y en Linux también), hay dos mecanismos distintos para localizar Java:

- **El `PATH` del sistema** decide qué `java.exe` se ejecuta cuando escribes `java` en consola.
- **La variable `JAVA_HOME`** es la que leen herramientas como Maven, Gradle, IDEs, scripts de build.

Si las dos apuntan a JDKs distintos, tienes una bomba de relojería:

- En consola: usas Java 11.
- Maven al compilar: usa Java 18.
- Eclipse o IntelliJ: usan lo que diga su configuración interna, que puede ser un tercer JDK distinto.

El proyecto puede compilar bien y romperse al ejecutar. O al revés. Y los mensajes de error son crípticos: `UnsupportedClassVersionError`, `class file has wrong version 61.0, should be 55.0`, etc. Detrás de esos errores hay siempre un JDK distinto del que crees.

**La lección:** antes de empezar cualquier proyecto Java, verificar que `java --version`, `javac --version`, `mvn --version` y el JDK del IDE **dicen exactamente lo mismo**. Si no, lo primero es alinearlos.

## El cuarto problema: el shim de Oracle

En la misma máquina apareció esto:

```
$ where java
C:\Program Files\Microsoft\jdk-11.0.16.101-hotspot\bin\java.exe
C:\Program Files\Common Files\Oracle\Java\javapath\java.exe
```

Dos `java.exe` en el `PATH`. El segundo es el famoso **shim de Oracle**: una carpeta vacía con accesos directos que Oracle instala con el JRE público y que apunta a "el último Java que has tenido instalado de Oracle". Suele ser un Java 8 antiguo, o nada, dependiendo de qué te haya pasado por la máquina a lo largo de los años.

¿Por qué importa? Porque si por cualquier motivo la primera entrada del PATH desaparece o cambia, te quedas con un Java 8 ejecutándose sin que lo sepas. Y todo deja de funcionar de formas creativas.

**La lección operativa**: en un entorno de desarrollo, mantén un solo JDK referenciado, controla el `PATH`, controla `JAVA_HOME`, y elimina shims que no usas.

## El quinto problema: el material desactualizado

El material original de este curso, preparado por otra persona, hablaba de:

- `javax.persistence` (muerto desde 2020).
- Apache Derby como driver (anacrónico, casi nadie lo usa fuera de tutoriales).
- MySQL en `localhost:3306` con `persistence.xml` cargado de propiedades.
- JUnit 4.11.

Si lo enseñáramos tal cual, estaríamos enseñando **un stack de hace 8 años**. El alumno saldría del curso con conocimientos que no le sirven en ninguna empresa actual.

Esto pasa **constantemente** con material de formación. Las tecnologías evolucionan, los materiales no. Hay tutoriales en YouTube con un millón de visitas que enseñan código que ya no compila. Hay cursos vendidos en plataformas grandes que siguen usando `javax.*`.

**Lección final del bloque:** desconfía de cualquier material (incluido el de este curso) que no diga claramente:

- Qué versión de Java usa.
- Qué versión de Hibernate usa.
- Si usa `javax` o `jakarta`.
- Cuándo se actualizó por última vez.

## Lo que vamos a hacer en este curso

Después de toda esta exposición de problemas, vamos a fijar el entorno con criterio:

| Componente | Versión | Justificación |
|---|---|---|
| **JDK** | 17 o superior | Necesario para Hibernate moderno |
| **Hibernate** | 6.4.x | Versión estable y reciente con `jakarta.*` |
| **JPA API** | `jakarta.persistence` 3.x | El estándar vivo |
| **Base de datos** | H2 en memoria | Cero fricción, recreable en cada arranque |
| **Build** | Maven | Más extendido en formación |
| **JUnit** | (no se usa, fuera del scope) | El curso no cubre testing |

Con esto fuera del camino, en el siguiente apartado entramos en la comparativa de versiones a fondo.

## Resumen ejecutivo

Cinco cosas que has aprendido en este apartado y que te van a perseguir el resto de tu vida profesional con Java:

1. **Las versiones importan.** Java, Hibernate y JPA tienen tablas de compatibilidad estrictas. No son intercambiables.
2. **`javax` ≠ `jakarta`.** Son APIs distintas con el mismo nombre de clases. Importar mal es un error de compilación o, peor, de runtime.
3. **`PATH` y `JAVA_HOME` deben coincidir.** Si no, errores aleatorios garantizados.
4. **Los materiales viejos enseñan stacks muertos.** Verifica fechas y versiones antes de seguir cualquier tutorial.
5. **El entorno se decide al principio del proyecto, no se descubre durante el desarrollo.** Cinco minutos de pensar las versiones te ahorran semanas de bugs.
