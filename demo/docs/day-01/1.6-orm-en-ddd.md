# 1.6. El sentido del ORM en DDD

## Por qué este apartado existe

Hasta ahora hemos hablado del ORM como **una herramienta técnica** que resuelve el *impedance mismatch*. Bien. Pero un ORM no se usa en el vacío: se usa en una aplicación que tiene una arquitectura, un modelo de dominio, unas reglas de negocio. Y aquí es donde el ORM se vuelve **una decisión arquitectónica**, no solo técnica.

Este apartado plantea **cómo encaja un ORM en una arquitectura DDD seria**, y por qué la decisión que tomamos al respecto define la calidad del código que vais a escribir el resto de vuestra carrera.

No vamos a explicar DDD desde cero (eso es un curso entero aparte). Vamos a aplicar sus principios mínimos a la pregunta: **¿dónde van las anotaciones de JPA?**

---

## Qué es DDD en una frase

**Domain-Driven Design** es una manera de diseñar software que pone el modelo de negocio en el centro:

- El **dominio** es lo importante. Todo lo demás (UI, base de datos, frameworks) son detalles de infraestructura.
- El **lenguaje del dominio** debe quedar reflejado en el código. Si los expertos del negocio hablan de "pizzas con ingredientes", el código tiene clases `Pizza` e `Ingrediente`, no `PizzaDto`, `PizzaEntity`, `PizzaModel`, `PizzaTO`...
- Las **reglas de negocio viven en el dominio**, no esparcidas por servicios y controladores.

Hay más cosas (agregados, value objects, contextos delimitados...) pero con esto basta para empezar.

---

## La pregunta clave: ¿dónde van las anotaciones de JPA?

Modelas el dominio. Tienes una clase `Pizza` con un método `calcularPrecio()` que aplica reglas de negocio reales. Llega el momento de persistirla. Tienes que decidir:

> ¿Pongo `@Entity`, `@Id`, `@Column` y `@OneToMany` directamente en `Pizza`, o las pongo en una clase distinta?

Hay tres respuestas posibles y cada una tiene consecuencias profundas.

---

## Respuesta 1: anotar la clase del dominio (lo que hace casi todo el mundo)

Es el camino fácil. Una sola clase. Misma `Pizza` con anotaciones JPA.

```java
@Entity
@Table(name = "pizzas")
public class Pizza {

    @Id
    @GeneratedValue
    private Long id;

    @Column(nullable = false)
    private String name;

    @Column
    private String description;

    @Column
    private String url;

    @ManyToMany
    @JoinTable(name = "pizza_ingredientes", ...)
    private List<Ingrediente> ingredientes = new ArrayList<>();

    @OneToMany(mappedBy = "pizza")
    private List<Comment> comments = new ArrayList<>();

    // Constructor sin args (¡obligatorio para JPA!)
    public Pizza() {}

    public BigDecimal calcularPrecio() {
        // ... lógica de negocio
    }
}
```

### Lo que parece bueno

- **Una sola clase, sencilla**.
- Menos código.
- Menos jerarquías que mantener.
- Es lo que enseñan todos los tutoriales.

### Lo que está roto (y nadie te cuenta)

#### 1. JPA exige cosas que rompen tu dominio

- **Constructor público sin argumentos.** Tu `Pizza` ya no puede garantizar invariantes en construcción. Cualquiera puede hacer `new Pizza()` y obtener una pizza inválida.
- **Setters públicos.** JPA necesita modificar atributos. Tu dominio queda anémico: cualquiera puede mutar el estado por fuera de los métodos de negocio.
- **Atributos no `final`.** No puedes tener inmutabilidad real.
- **Listas inicializadas a colecciones vacías.** Si no, JPA puede meterte un `null` y tu dominio explota.

#### 2. El dominio queda atado a la infraestructura

Tu `Pizza` ya no es un POJO. Para compilar, necesita el JAR de `jakarta.persistence` en el classpath. Eso significa:

- **No puedes testar el dominio sin Hibernate**.
- **No puedes mover el dominio a otra capa** sin arrastrar dependencias de persistencia.
- **No puedes cambiar de Hibernate a otra cosa** (jOOQ, MyBatis, repositorio en memoria, base de datos NoSQL) sin reescribir el dominio.

#### 3. Las decisiones de carga contaminan el modelo

En el momento en que pones `@OneToMany(fetch = LAZY)` en `Pizza.comments`, has tomado una decisión de **rendimiento de base de datos** dentro de tu modelo de negocio. Esa decisión no es de negocio. Y, peor, depende del caso de uso: en unos sí quieres cargar los comentarios, en otros no.

#### 4. Cargar implica un proxy mágico

`pizza.getComments()` no es una llamada normal a un getter. Si `Pizza` es una entidad gestionada por Hibernate, ese método **puede ejecutar una query SQL en cualquier momento**. Tu dominio ya no se comporta de forma predecible: el coste de un getter depende del contexto.

#### 5. `equals` y `hashCode` se vuelven un drama

El dominio quiere `equals` por identidad de negocio (dos pizzas Margarita con mismos ingredientes son iguales). JPA quiere `equals` por id de BD. Las dos versiones rompen casos de uso reales. No hay una solución limpia.

#### 6. Errores en runtime

`LazyInitializationException` cuando accedes a una colección fuera de la sesión. Tu dominio "decide" cargar algo y revienta porque no hay sesión abierta. La excepción no tiene sentido **en el modelo de dominio**: solo lo tiene en el modelo de persistencia.

### El veredicto

Esta opción **funciona**. La mayoría del software del mundo está hecho así. Pero:

- No es DDD. Es DDD con anotaciones encima fingiendo serlo.
- Acumula deuda técnica.
- En proyectos pequeños o de corto plazo, es razonable.
- En proyectos largos o con reglas de negocio complejas, es una bomba de relojería.

## Respuesta 2: modelo anémico (el "punto medio" peligroso)

La gente que se da cuenta de los problemas anteriores a veces toma este atajo: hacer las clases JPA puros sacos de getters y setters, y poner toda la lógica de negocio en clases `Service`.

```java
@Entity
public class Pizza {
    @Id Long id;
    String name;
    // ... solo datos, getters/setters
}

public class PizzaService {
    public BigDecimal calcularPrecio(Pizza pizza) { ... }
    public void añadirIngrediente(Pizza pizza, Ingrediente i) { ... }
    public boolean esVegana(Pizza pizza) { ... }
}
```

### Lo que parece bueno

- Separa datos de comportamiento.
- "Más limpio" a primera vista.
- Es lo que muchos proyectos Spring acaban siendo.

### Lo que en realidad pasa

Esto es lo que Martin Fowler llamó ***Anemic Domain Model*** y lo definió como un **antipatrón**. Lo nombró así para criticarlo, no para alabarlo.

¿Por qué es malo?

- **No es orientación a objetos.** Es procedural disfrazado. Tienes "estructuras de datos" (las entidades) y "funciones que actúan sobre ellas" (los servicios). Eso es C, no Java.
- **La lógica se dispersa.** ¿Dónde está la regla "una pizza no puede tener menos de un ingrediente"? En `PizzaService`, en `IngredienteService`, en `PedidoService`... y tarde o temprano la regla está en tres sitios distintos con tres versiones distintas.
- **Las invariantes no se protegen.** Como `Pizza` no tiene constructor que valide, cualquiera puede crear pizzas inválidas. La validación queda a cargo del programador que se acuerde de llamar al servicio correcto.
- **Pierdes el lenguaje del dominio.** Los expertos hablan de "pizzas que tienen un precio". Tu código tiene "estructuras Pizza" y "servicios que les calculan el precio". Ya no hablas el mismo idioma.

### El veredicto

**Es peor que la opción 1**. Tiene los inconvenientes (modelo atado a JPA) sin los beneficios (al menos en la opción 1, la lógica está en la entidad y se respeta el lenguaje del dominio).

Esta opción surge cuando alguien ha leído "evita poner lógica en las entidades JPA" sin entender por qué, y aplica el remedio incorrecto. **No la elijáis nunca conscientemente**.

---

## Respuesta 3: dominio puro + entidad de persistencia + mapper + repository

Esta es **la opción que vamos a usar en el curso**. Es más cara, más correcta y la que tiene sentido cuando se hace DDD en serio.

### La idea

Tienes **dos modelos paralelos**, en capas distintas:

```
domain/
    Pizza.java           ← POJO puro. Constructores que validan.
    Ingrediente.java        Métodos de negocio. Sin anotaciones JPA.
    Comment.java
    User.java
    PizzaRepository.java  ← Interfaz. Solo el contrato.

infrastructure/persistence/
    PizzaEntity.java     ← Con @Entity, @Id, @OneToMany...
    IngredienteEntity.java   POJO técnico para JPA. Setters públicos.
    CommentEntity.java       Constructor sin args.
    UserEntity.java
    PizzaMapper.java     ← Traduce Pizza ↔ PizzaEntity
    PizzaRepositoryJpa.java ← Implementación con EntityManager
```

### Cómo encajan las piezas

#### `domain.Pizza` (el dominio)

```java
public final class Pizza {

    private final PizzaId id;
    private final String name;
    private final String description;
    private final List<Ingrediente> ingredientes;

    public Pizza(PizzaId id, String name, List<Ingrediente> ingredientes) {
        if (name == null || name.isBlank())
            throw new IllegalArgumentException("Pizza requiere nombre");
        if (ingredientes == null || ingredientes.isEmpty())
            throw new IllegalArgumentException("Pizza requiere al menos un ingrediente");

        this.id = id;
        this.name = name;
        this.description = null;
        this.ingredientes = List.copyOf(ingredientes);
    }

    public BigDecimal calcularPrecio() {
        BigDecimal costeIngredientes = ingredientes.stream()
            .map(Ingrediente::getCost)
            .reduce(BigDecimal.ZERO, BigDecimal::add);
        return costeIngredientes.multiply(new BigDecimal("1.20"));
    }

    public boolean contieneIngrediente(IngredienteId id) {
        return ingredientes.stream().anyMatch(i -> i.getId().equals(id));
    }

    // Getters, equals/hashCode por identidad de negocio
}
```

Esta clase:
- **No tiene anotaciones JPA**.
- **No conoce Hibernate**.
- **Compila sin Hibernate en el classpath**.
- Tiene el constructor que valida invariantes.
- Es inmutable (atributos `final`).
- Tiene comportamiento real.
- Es testeable sin tocar BD.

#### `infrastructure.persistence.PizzaEntity` (la entidad JPA)

```java
@Entity
@Table(name = "pizzas")
public class PizzaEntity {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(name = "name", nullable = false, length = 100)
    private String name;

    @Column(name = "description", length = 1000)
    private String description;

    @ManyToMany(fetch = FetchType.LAZY)
    @JoinTable(name = "pizza_ingredientes", ...)
    private List<IngredienteEntity> ingredientes = new ArrayList<>();

    public PizzaEntity() {}  // exigido por JPA

    // setters públicos, sin invariantes
}
```

Esta clase:
- **Es un POJO técnico para JPA**.
- Tiene constructor sin args, setters públicos.
- No tiene lógica de negocio.
- Vive en la capa de infraestructura.
- Cambia con frecuencia (esquema de BD, fetch strategies).

#### `infrastructure.persistence.PizzaMapper` (el traductor)

```java
public class PizzaMapper {

    public Pizza toDomain(PizzaEntity entity) {
        List<Ingrediente> ingredientes = entity.getIngredientes().stream()
            .map(IngredienteMapper::toDomain)
            .toList();

        return new Pizza(
            new PizzaId(entity.getId()),
            entity.getName(),
            ingredientes
        );
    }

    public PizzaEntity toEntity(Pizza pizza) {
        PizzaEntity entity = new PizzaEntity();
        entity.setId(pizza.getId().value());
        entity.setName(pizza.getName());
        // ...
        return entity;
    }
}
```

#### `domain.PizzaRepository` (el contrato)

```java
public interface PizzaRepository {
    Optional<Pizza> findById(PizzaId id);
    List<Pizza> findAll();
    void save(Pizza pizza);
    void delete(PizzaId id);
}
```

#### `infrastructure.persistence.PizzaRepositoryJpa` (la implementación)

```java
public class PizzaRepositoryJpa implements PizzaRepository {

    private final EntityManager em;
    private final PizzaMapper mapper;

    public PizzaRepositoryJpa(EntityManager em, PizzaMapper mapper) {
        this.em = em;
        this.mapper = mapper;
    }

    @Override
    public Optional<Pizza> findById(PizzaId id) {
        PizzaEntity entity = em.find(PizzaEntity.class, id.value());
        return Optional.ofNullable(entity).map(mapper::toDomain);
    }

    @Override
    public void save(Pizza pizza) {
        PizzaEntity entity = mapper.toEntity(pizza);
        em.merge(entity);
    }

    // ...
}
```

### Lo que ganas

#### 1. El dominio es **portable**

Compila sin Hibernate. Funciona sin Hibernate. Si mañana decides usar MongoDB, jOOQ o un repositorio en memoria, **tu dominio no cambia una línea**. Solo cambias la implementación del repositorio.

#### 2. Las invariantes están **protegidas por construcción**

Es **imposible** crear una `Pizza` inválida desde fuera. El constructor lo prohíbe.

#### 3. Las decisiones de carga **viven en el repositorio**, no en el dominio

`Pizza` no tiene `List<Comment>`. ¿Quieres los comentarios de una pizza? Llamas a `commentRepository.findByPizzaId(pizzaId)`. Decisión explícita, controlada, sin proxies mágicos.

#### 4. Las invariantes de negocio y las restricciones de almacenamiento **se separan**

- "Una pizza requiere al menos un ingrediente" → invariante de dominio. Vive en el constructor de `Pizza`.
- "La columna `name` es `VARCHAR(100) NOT NULL`" → restricción de almacenamiento. Vive en `@Column` de `PizzaEntity`.

Los dos sitios. Los dos contextos. **No es duplicación: es protección en capas distintas**.

#### 5. Tu dominio se vuelve **testeable de verdad**

Tests unitarios de `Pizza.calcularPrecio()` sin levantar contexto Spring, sin tocar BD, sin mocks de `EntityManager`. Solo POJOs.

#### 6. Los `LazyInitializationException` **no existen**

El dominio no tiene proxies. No tiene sesiones. No puede explotar por intentar cargar algo fuera de contexto.

### Lo que pagas

Sé honesto: esto **cuesta más**.

- **Doble jerarquía**: cuatro entidades de dominio + cuatro entidades JPA + cuatro mappers.
- **Más código por feature**: añadir un campo significa tocarlo en `Pizza`, en `PizzaEntity` y en `PizzaMapper`.
- **Más decisiones explícitas**: ¿el repositorio devuelve agregados completos? ¿proyecciones específicas? ¿paginación?
- **Curva de aprendizaje**: equipos junior se pierden en las capas si no hay quien guíe.
- **Mapeo manual o con MapStruct**: en proyectos grandes, MapStruct (generación de mappers en compilación) salva la vida.

### Cuándo elegir esta opción

- Proyectos con **reglas de negocio complejas**.
- Proyectos con **vida larga** (cinco años o más).
- Proyectos con **alta probabilidad de cambiar de tecnología de persistencia**.
- Proyectos donde el **dominio es el activo de negocio**, no un detalle de implementación.
- Equipos con disciplina y madurez para mantener las capas separadas.

### Cuándo NO elegirla

- CRUDs sencillos, aplicaciones pequeñas, prototipos.
- Equipos sin experiencia con la separación de capas.
- Plazos cortos donde el dominio no es lo crítico.

---

## La decisión arquitectónica del curso

En este curso vamos a usar **la respuesta 3**. Por dos razones:

1. **Es lo correcto para el modelo que tenemos.** Pizzas, ingredientes, comentarios, usuarios. Hay reglas de negocio (cálculo del precio, agregados, validaciones) que merecen un dominio limpio.

2. **Es la que enseña a pensar.** Con la opción 1, aprendes a anotar clases. Con la opción 3, aprendes a tomar decisiones arquitectónicas. La diferencia entre saber JPA y saber arquitectura.

Y también vamos a ser honestos con vosotros: **en muchas empresas vais a encontrar la opción 1**. No por mejor, sino por inercia, por velocidad, por equipos que no conocen la alternativa. Saber por qué es problemática y cuándo merece la pena el esfuerzo de la opción 3 es lo que os va a diferenciar.

---

## Resumen

- **DDD pone el modelo de negocio en el centro.** Todo lo demás es infraestructura.
- Hay tres formas de combinar JPA con DDD:
  1. **Anotar la entidad de dominio** (rápido, popular, contamina el dominio).
  2. **Modelo anémico con servicios** (antipatrón, peor de lo que parece).
  3. **Dominio puro + entidad JPA + mapper + repository** (correcto, costoso).
- En este curso elegimos la opción 3 con todas sus consecuencias.
- El ORM no se "pelea" con DDD: el problema es **dónde se ponen sus anotaciones**.
- Las invariantes de negocio viven en el dominio. Las restricciones de almacenamiento viven en la entidad JPA. **No es duplicación, es protección en capas distintas.**
- Esta decisión tiene coste real (más código, más capas) y beneficio real (dominio portable, testeable, protegido). Hay que elegirla con criterio.
