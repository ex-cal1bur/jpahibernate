# 2.5. Agregados: qué entra dentro, qué queda fuera

## El concepto que falta en casi todos los modelos

Cuando alguien dibuja "Pizza tiene Ingredientes y Comments, Comment tiene User", lo natural es pensar en un **grafo plano** de entidades relacionadas. Todo apunta a todo, todo se navega desde todo. Es lo que enseñan los tutoriales clásicos de "modelado entidad-relación".

DDD introduce un concepto que cambia el dibujo: el **agregado**. Y entender qué es y para qué sirve es lo que separa un modelo limpio de un modelo que **acaba siendo un nudo de relaciones bidireccionales con problemas de carga**, ciclos en `toString`, transacciones que tocan media base de datos y bugs raros de consistencia.

---

## Qué es un agregado

Un **agregado** es un grupo de entidades que se tratan como **una unidad** para tres cosas:

1. **Consistencia**: las invariantes del agregado se mantienen siempre como un todo.
2. **Persistencia**: el agregado se guarda y se carga de forma atómica.
3. **Acceso**: solo se entra al agregado por su **raíz** (`aggregate root`).

Ejemplo: una `Pizza` con sus `Ingredientes`. Si decís que es un agregado:

- Una pizza con cero ingredientes no es válida (invariante del agregado).
- Cuando guardáis la pizza, guardáis todos sus ingredientes con ella.
- Nadie modifica los ingredientes de una pizza pasando por encima de la pizza. **Todo va por `Pizza`**.

---

## Por qué importa el "dentro/fuera" del agregado

Volvamos al modelo:

- `Pizza`
- `Ingrediente`
- `Comment`
- `User`

Si lo dibujáis con líneas:

```
        Pizza ── Ingrediente
          │
          │
        Comment ── User
```

Pregunta: **¿qué pertenece al agregado de `Pizza` y qué no?**

Y otra pregunta más sutil: **¿hay uno, dos, tres o cuatro agregados aquí?**

Las respuestas no son obvias. Vamos por partes.

---

## Decisión 1: ¿`Ingrediente` está dentro del agregado `Pizza`?

Hay dos opciones, y ambas se ven en proyectos reales.

### Opción A — `Ingrediente` es **parte interna** de `Pizza`

```java
public class Pizza {
    private final List<Ingrediente> ingredientes;
    // ...
}
```

Implicaciones:

- `Ingrediente` no existe sin una `Pizza` que lo contenga.
- Un mismo "queso mozzarella" se duplica en cada pizza que lo lleva.
- Si cambia el coste de la mozzarella, hay que actualizarlo en **todas las pizzas**.
- Cuando guardamos una pizza, se guardan sus ingredientes con ella.
- No hay un repositorio de ingredientes; solo se accede a través de las pizzas.

¿Cuándo tiene sentido? Cuando los ingredientes son **parte de la receta**, no entidades independientes del catálogo. Si la pizzería piensa "cada pizza tiene sus ingredientes y se acabó", esto es razonable.

### Opción B — `Ingrediente` es entidad **independiente**, y `Pizza` solo la **referencia**

```java
public class Pizza {
    private final List<IngredienteId> ingredientes; // referencias por id
    // o, según convenio:
    // private final List<Ingrediente> ingredientes; // pero lazy-loaded
    // ...
}

public class Ingrediente {
    // existe independientemente; vive en su propio repositorio
}
```

Implicaciones:

- Hay un **catálogo de ingredientes** centralizado: `IngredienteRepository`.
- "Mozzarella" es **un solo objeto** referenciado por las pizzas que la usan.
- Si cambia el coste de la mozzarella, se actualiza **una vez** y todas las pizzas reflejan el nuevo coste.
- Pizza e Ingrediente son **agregados separados**.
- Para crear una pizza, primero existen los ingredientes; se referencian por id.

¿Cuándo tiene sentido? Cuando los ingredientes son **catálogo**, no parte de la receta. Si la pizzería piensa "tengo mis ingredientes con sus costes, y construyo pizzas con ellos", esto es lo correcto.

### Cuál elegimos en este curso

**Opción B**: `Ingrediente` es entidad independiente. `Pizza` lo referencia.

Razones:

- Refleja mejor el negocio (un catálogo de ingredientes existe).
- El cálculo del precio depende del coste **actual** del ingrediente, no de una copia congelada en cada pizza.
- Cuando lleguemos al bloque 3, esto se traduce a una `@ManyToMany` clásica entre `pizzas` e `ingredientes`, que es buen ejercicio didáctico.

Importante: **eso significa que tenemos al menos dos agregados**: el de `Pizza` y el de `Ingrediente`.

---

## Decisión 2: ¿`Comment` está dentro del agregado `Pizza`?

Volvemos a tener dos opciones razonables.

### Opción A — `Comment` es **parte de** `Pizza`

```java
public class Pizza {
    private final List<Comment> comments;

    public void añadirComentario(Comment c) { ... }
}
```

- Para añadir un comentario, hay que pasar por la pizza.
- `pizza.getComments()` devuelve la lista.
- Si se borra la pizza, se borran sus comentarios.

### Opción B — `Comment` es **agregado independiente** que **referencia** a la pizza

```java
public class Comment {
    private final PizzaId pizzaId; // qué pizza comenta
    private final UserId userId;   // quién la comenta
    // ...
}

public class Pizza {
    // no conoce sus comentarios
}
```

- Hay un `CommentRepository`.
- `Pizza` no sabe nada de sus comentarios.
- Si quieres "los comentarios de la pizza X", consultas al `CommentRepository`.

### Cuál elegimos

**Opción B**: `Comment` es agregado independiente.

Razones técnicas y de modelado:

- Una pizza puede tener **miles de comentarios** con el tiempo. Si los metiéramos dentro del agregado, **cargar una pizza implicaría cargar todos sus comentarios**. Catastrófico en cuanto el catálogo crece.
- Los comentarios **se consultan mayoritariamente solos** (ordenados por fecha, paginados, filtrados por usuario), no junto a la pizza.
- Borrar una pizza no implica necesariamente borrar sus comentarios (puede haber históricos, auditoría).
- La consistencia transaccional entre pizza y comentarios no es crítica: añadir un comentario no necesita que la pizza esté en la misma transacción.

Esta decisión es **muy importante**. Cuando lleguemos a JPA y al bloque 4 (carga lazy/eager, agregado completo vs lazy en colecciones), esta decisión se va a traducir a:

- **Ingredientes** (parte conceptual de la pizza) → carga **eager** o explícita al cargar la pizza.
- **Comentarios** (fuera del agregado) → carga **lazy**, se consultan aparte.

Si esto no se decide aquí, en el bloque 4 lo decidiremos a martillazos contra `LazyInitializationException`.

---

## Decisión 3: ¿`User` y `Comment` van juntos?

`Comment` apunta a `User`. ¿Es `User` parte del agregado `Comment`?

**No**. Razones obvias:

- Un `User` existe independientemente de cualquier comentario que haya escrito.
- Un `User` puede tener muchos comentarios; agruparlo dentro de uno no tendría sentido.
- `User` tiene su propio ciclo de vida (alta, baja, modificación) totalmente separado del de los comentarios.

`Comment` simplemente **referencia** al `User` por su id. Cuando hace falta cargar el usuario completo, se pasa por el `UserRepository`.

---

## El mapa de agregados del curso

Después de las tres decisiones:

```
Agregado 1: Pizza
   - Pizza (raíz)
   - referencia → IngredienteId (o Ingrediente como referencia)

Agregado 2: Ingrediente
   - Ingrediente (raíz)

Agregado 3: Comment
   - Comment (raíz)
   - referencia → PizzaId
   - referencia → UserId

Agregado 4: User
   - User (raíz)
```

Cuatro agregados. Cuatro raíces. Cuatro repositorios (uno por raíz).

```
PizzaRepository
IngredienteRepository
CommentRepository
UserRepository
```

Y **ningún** `IngredienteRepository.añadirAPizza(...)` ni cosas raras: cada repositorio gestiona **su raíz**. Las relaciones entre agregados se manejan **por id**, no por navegación directa.

---

## Reglas de oro de los agregados

1. **El acceso al agregado es solo por la raíz.** Nadie llega a un `Comment` saltándose el `CommentRepository`. Nadie modifica un `Ingrediente` desde una `Pizza`.

2. **Las referencias entre agregados son por id**, no por instancia. `Comment` tiene `PizzaId` y `UserId`, no `Pizza` y `User`. Esto evita cargar todo el universo cuando consultas una sola cosa.

3. **Una transacción modifica un solo agregado**. Si una operación de negocio necesita cambiar dos agregados a la vez, **probablemente tu modelo está mal** (o necesita gestionarse con eventos de dominio, que es tema avanzado).

4. **Las invariantes viven en el agregado**. La regla "una pizza tiene al menos un ingrediente" es invariante del agregado `Pizza`. La regla "un comentario tiene un autor" es invariante del agregado `Comment`. Ningún agregado es responsable de las invariantes de otro.

5. **El tamaño del agregado debe ser pequeño**. Un agregado con cien colecciones grandes es una mala señal. Pizza con sus ingredientes (pocos) es razonable. Pizza con sus diez mil comentarios sería un desastre.

---

## Por qué esto importa muchísimo cuando llega JPA

Ahora viene la parte que justifica retroactivamente todo lo dicho.

Cuando en el bloque 3 anotemos las entidades de persistencia, **cada decisión de agregado se traduce a una decisión técnica**:

| Decisión de agregado | Traducción JPA |
|---|---|
| Ingrediente vive **dentro** del agregado Pizza | Carga **eager** o `JOIN FETCH` al cargar Pizza |
| Comment vive **fuera** del agregado Pizza | Carga **lazy** o no relacionado en absoluto |
| Comment **referencia** a User por id | `@ManyToOne` con la opción de cargar User aparte |
| User es agregado independiente | `UserRepository` con queries propias |

Y en el bloque 4 (decisiones de carga, lazy vs eager), la regla acaba siendo **muy simple**:

> **Lo que está dentro del agregado se carga con el agregado. Lo que está fuera, lazy o por consulta separada.**

Si no habéis decidido qué está dentro y qué está fuera, **no podéis decidir qué cargar y qué no**. Vais a tener `FetchType.EAGER` por defecto en todo y pagar por ello con N+1, cargas masivas y `LazyInitializationException`.

**Las decisiones de hoy son las que evitan los bugs del bloque 5.**

## Resumen

- Un **agregado** es un grupo de entidades que se trata como una unidad de **consistencia**, **persistencia** y **acceso**.
- Cada agregado tiene una **raíz** (entity root). Solo se accede al agregado por la raíz.
- **Las relaciones entre agregados son por id**, no por instancia. `Comment` tiene `PizzaId`, no `Pizza`.
- En el modelo de la pizzería tenemos **cuatro agregados**: Pizza, Ingrediente, Comment, User. Cada uno con su repositorio.
- `Ingrediente` se eligió como **agregado independiente** (catálogo centralizado), no como parte interna de Pizza.
- `Comment` se eligió como **agregado independiente** (no interno a Pizza) por **rendimiento** (los comentarios pueden ser muchos) y por **modelo** (se consultan aparte).
- Las decisiones de agregado tomadas hoy se traducirán mañana en decisiones técnicas: **qué cargar eager, qué cargar lazy, qué meter en una transacción, qué borrar en cascada**.
- **Sin decisión de agregado, no hay decisión de carga**. Por eso este apartado va antes de JPA, no después.
