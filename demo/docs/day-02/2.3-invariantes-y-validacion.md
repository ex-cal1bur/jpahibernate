# 2.3. Invariantes y validación en el dominio

## La pregunta que resuelve este apartado

> **¿Dónde vive la regla "una pizza siempre tiene al menos un ingrediente"?**

Hay tres respuestas habituales en la vida real:

1. En el `PizzaController`, validando el JSON que llega.
2. En un `PizzaValidator` que se invoca desde el `PizzaService`.
3. En la propia clase `Pizza`, dentro del factory `create` y de los métodos de mutación.
4. *Bonus*: en `@NotEmpty` sobre el campo de la entidad JPA.

Las cuatro **se ven** en proyectos reales. Solo una protege el dominio. Vamos a ver por qué.

---

## Qué es una invariante

Una **invariante** es una propiedad que **siempre** tiene que ser verdad mientras un objeto exista. No "casi siempre". No "después de validar". **Siempre**.

Ejemplos en nuestro modelo:

- Una `Pizza` siempre tiene un nombre no vacío.
- Una `Pizza` siempre tiene al menos un ingrediente.
- Un `Ingrediente` siempre tiene un coste mayor o igual a cero.
- Un `Comment` siempre tiene un autor.
- Un `Comment` siempre tiene una fecha **no futura**.
- Un `User` siempre tiene un email con formato razonable.

Si en algún momento de la vida del objeto **alguno de estos invariantes deja de cumplirse**, el objeto está roto. Y si está roto, todo el código que dependa de él va a comportarse mal de formas impredecibles.

> **Importante**: las invariantes se validan **cuando el estado se crea o cambia**, no cuando se reconstruye desde persistencia. El factory `restore` confía en que los datos guardados ya eran válidos en su día. Lo vimos en 2.2.

---

## Dónde NO van las invariantes

### En el controlador

```java
@PostMapping("/pizzas")
public ResponseEntity<?> crear(@RequestBody PizzaDto dto) {
    if (dto.getName() == null || dto.getName().isBlank())
        return ResponseEntity.badRequest().build();
    if (dto.getIngredientes() == null || dto.getIngredientes().isEmpty())
        return ResponseEntity.badRequest().build();
    Pizza pizza = Pizza.create(dto.getName(), dto.getIngredientes());
    // ...
}
```

Lo que parece bueno: filtra entradas malas antes de tocar nada importante.

Lo que está roto: la validación **solo cubre este punto de entrada**. Si mañana añadimos:

- Un endpoint diferente que también crea pizzas.
- Un job batch que importa pizzas desde un CSV.
- Un test que crea pizzas directamente.
- Un script de migración.

…cada uno tiene que **reimplementar** la validación. Y antes o después, alguno se la va a saltar. Y va a haber pizzas inválidas en producción.

> **La regla del controlador**: el controlador valida **el formato** (¿es un JSON bien formado? ¿el campo es un string?). El **dominio** valida **la regla de negocio**.

### En un Validator externo

```java
public class PizzaValidator {
    public void validar(Pizza p) {
        if (p.getName() == null || p.getName().isBlank())
            throw new ValidationException("nombre obligatorio");
        // ...
    }
}
```

Lo que parece bueno: una sola responsabilidad, un punto único de validación.

Lo que está roto: para que esto funcione, **alguien tiene que acordarse de llamarlo**. Si en algún flujo se olvida, se cuela una pizza inválida. La validación deja de ser una invariante (siempre verdadera) y pasa a ser una **convención** (verdadera si todo el equipo se acuerda).

> **La regla del Validator**: el `Validator` es **opcional**, lo que está mal. Las invariantes son **obligatorias**.

### En `@NotEmpty` sobre la entidad JPA

```java
@Entity
public class PizzaEntity {
    @NotBlank
    private String name;

    @NotEmpty
    @ManyToMany
    private List<IngredienteEntity> ingredientes;
}
```

Lo que parece bueno: queda registrado en el código, lo aplica el framework.

Lo que está roto: se ejecuta solo **al persistir**. Es decir, **demasiado tarde**:

- La pizza ya se creó en memoria.
- La lógica de negocio (`calcularPrecio`) ya se ejecutó (con datos rotos).
- Otros métodos ya pudieron usarla.
- Si el flujo no llega a `persist()`, la validación **nunca se ejecuta**.

Bean Validation en la entidad JPA tiene su sitio (lo veremos en 3.8), pero **no como sustituto** de las invariantes del dominio. Es otra capa, no la única.

## Dónde SÍ van las invariantes: en el factory `create` y en los métodos de mutación

La única respuesta correcta:

```java
public class Ingrediente {

    public static Ingrediente create(String description, Money cost) {
        validarDescription(description);
        validarCost(cost);
        return new Ingrediente(IngredienteId.nuevo(), description, cost);
    }

    public void renombrar(String description) {
        validarDescription(description);   // se vuelve a validar
        this.description = description;
    }

    public void actualizarCoste(Money cost) {
        validarCost(cost);                  // se vuelve a validar
        this.cost = cost;
    }

    private static void validarDescription(String description) {
        if (description == null || description.isBlank())
            throw new IllegalArgumentException("description obligatoria");
    }

    private static void validarCost(Money cost) {
        if (cost == null) throw new IllegalArgumentException("cost obligatorio");
        if (cost.isNegative()) throw new IllegalArgumentException("cost no puede ser negativo");
    }
}
```

¿Por qué es la única correcta?

- **No se puede crear un `Ingrediente` saltándose la validación.** El constructor es privado. La única forma de obtener uno es `create(...)`, que valida.
- **No se puede mutar un `Ingrediente` saltándose la validación.** No hay setters. La única forma de cambiar `description` es `renombrar(...)`, que vuelve a validar.
- **No depende de ningún framework**, ni de Spring, ni de JPA, ni de nadie.
- **Funciona en cualquier flujo**: REST, batch, tests, scripts. Da igual quién cree o modifique el ingrediente, las invariantes se cumplen.
- **Falla rápido**. En el momento del cambio, no más adelante cuando ya es tarde.
- **El compilador y el linter no tienen que recordar nada**. La invariante está en el sitio donde el estado entra o cambia.

---

## Una nota sobre `restore`

`restore` **no valida**. Lo justificamos en 2.2 pero conviene repetirlo aquí porque es contraintuitivo:

```java
public static Ingrediente restore(IngredienteId id, String description, Money cost) {
    return new Ingrediente(id, description, cost);
}
```

Si los datos están en BD, ya eran válidos cuando se guardaron. Volver a validarlos al cargarlos:

- Es redundante.
- Hace lenta la carga.
- **Rompe la aplicación si las invariantes cambian con el tiempo y los datos antiguos ya no las cumplen** (cosa legítima: la regla "el nombre debe tener al menos 3 caracteres" añadida en 2026 no debería invalidar usuarios creados en 2024 con nombre "Al").

La validación es responsabilidad del momento de **cambio de estado** (factory de creación o mutador), no del momento de **rehidratación**.

---

## Validación vs invariantes: la distinción importante

A veces se confunden, y conviene separarlos:

| Concepto | Qué es | Dónde vive |
|---|---|---|
| **Validación de entrada** | El JSON que llega al endpoint tiene los campos esperados, los tipos correctos, formato razonable. | Capa de entrada (controlador, parser). |
| **Invariante de dominio** | El objeto, una vez creado o modificado, cumple las reglas del negocio. | **Factory `create` y métodos de mutación de la entidad de dominio**. |
| **Constraint de almacenamiento** | La fila de la tabla cumple las restricciones del esquema (NOT NULL, unique, length). | Esquema de BD + anotaciones JPA. |

Las tres pueden coexistir. **No se duplican**, se complementan:

- El controlador rechaza un JSON sin el campo `name`. Mensaje de error claro al cliente.
- El dominio rechaza una `Pizza` con nombre vacío. Bug del programador si llega a este punto.
- La BD rechaza un `INSERT` con `name = NULL`. Última red de seguridad.

Cada capa protege contra el fallo de la anterior. Si una capa hace bien su trabajo, las siguientes casi nunca disparan. Pero si una falla, la siguiente lo coge.

## Cuando una invariante merece su propio value object: `FechaComentario`

Un caso real del modelo. La regla:

> Un `Comment` tiene una `fecha` **no nula** y **no futura** (no se puede comentar algo en el año 2050).

La forma ingenua de implementarlo:

```java
public class Comment {
    private final LocalDateTime fecha;

    public static Comment create(String description, UserId author) {
        LocalDateTime fecha = LocalDateTime.now();
        if (fecha == null || fecha.isAfter(LocalDateTime.now()))
            throw new IllegalArgumentException("fecha inválida");
        // ...
    }
}
```

Dos problemas:

1. La invariante "no futura" vive en `Comment`. Si mañana queremos una `FechaPedido` con la misma regla, **se duplica**.
2. `LocalDateTime` no transmite **qué es** esa fecha. ¿Es la creación? ¿El último cambio? ¿Una fecha de pedido? El compilador no lo sabe.

**Primitive obsession** de manual. La cura: envolverla en un value object propio.

```java
public record FechaComentario(LocalDateTime value) {

    public FechaComentario {
        if (value == null)
            throw new IllegalArgumentException("fecha obligatoria");
        if (value.isAfter(LocalDateTime.now()))
            throw new IllegalArgumentException("la fecha no puede ser futura");
    }

    public static FechaComentario ahora() {
        return new FechaComentario(LocalDateTime.now());
    }

    public boolean esAnteriorA(FechaComentario otra) {
        return this.value.isBefore(otra.value);
    }
}
```

Beneficios:

- **La invariante "no futura" vive aquí**, no en `Comment`. Donde tiene que vivir.
- **Imposible crear una `FechaComentario` futura**. El compact constructor lo prohíbe.
- **Tipo descriptivo**: `FechaComentario` cuenta qué es. `LocalDateTime` no.
- **Reutilizable**: si mañana otra entidad necesita una "fecha de evento que no sea futura", el patrón está claro.
- **Como es un `record`, lleva gratis** equals/hashCode por valor, toString legible, getters automáticos.
- **Encaja con Hibernate como `@Embeddable`** (lo veremos en 3.3).

Y `Comment` queda así:

```java
public class Comment {

    private final CommentId id;
    private final FechaComentario fecha;
    private String description;
    private final UserId author;

    public static Comment create(String description, UserId author) {
        validarDescription(description);
        validarAuthor(author);
        return new Comment(
            CommentId.nuevo(),
            FechaComentario.ahora(),
            description,
            author
        );
    }

    public static Comment restore(CommentId id, FechaComentario fecha,
                                   String description, UserId author) {
        return new Comment(id, fecha, description, author);
    }

    private Comment(CommentId id, FechaComentario fecha,
                    String description, UserId author) {
        this.id = id;
        this.fecha = fecha;
        this.description = description;
        this.author = author;
    }

    public void editar(String description) {
        validarDescription(description);
        this.description = description;
    }

    private static void validarDescription(String description) {
        if (description == null || description.isBlank())
            throw new IllegalArgumentException("description obligatoria");
    }

    private static void validarAuthor(UserId author) {
        if (author == null) throw new IllegalArgumentException("author obligatorio");
    }

    public CommentId getId() { return id; }
    public FechaComentario getFecha() { return fecha; }
    public String getDescription() { return description; }
    public UserId getAuthor() { return author; }

    @Override public boolean equals(Object o) {
        return o instanceof Comment other && this.id.equals(other.id);
    }
    @Override public int hashCode() { return id.hashCode(); }
}
```

Notad que `Comment` ya **no se ocupa de validar la fecha**. La fecha **se valida sola** porque su tipo lo garantiza. Esto es exactamente la lección de Fowler: **mover invariantes al tipo más pequeño que pueda contenerlas**.

## ¿Y si una validación necesita información externa?

A veces la regla no se puede comprobar mirando solo el objeto. Ejemplo: "no pueden existir dos `User` con el mismo email".

Eso **no es una invariante de la entidad**. Es una invariante del **agregado o del repositorio**. `User.create("juan@x.com")` no tiene forma de saber si ya existe otro usuario con ese email; eso lo sabe el `UserRepository`.

```java
public class CrearUsuarioUseCase {

    private final UserRepository repo;

    public User ejecutar(String name, String email) {
        if (repo.existePorEmail(email))
            throw new EmailYaRegistradoException(email);
        User u = User.create(name, email);   // formato del email se valida aquí
        repo.guardar(u);
        return u;
    }
}
```

División clara:

- **Formato** del email (contiene `@`, no es nulo) → invariante de `User`.
- **Unicidad** del email (no hay otro igual en BD) → responsabilidad del caso de uso, que consulta el repositorio.

Mezclar las dos cosas dentro de `User.create` obliga a inyectar el repositorio en la entidad de dominio, **lo cual contamina el dominio con infraestructura**. No se hace.

## ¿Qué pasa con Bean Validation (`@NotNull`, `@NotBlank`, `@Size`...)?

Bean Validation es **una herramienta de validación declarativa**, no una herramienta de invariantes. Sirve para:

- **Validar entradas en controladores**: `@Valid @RequestBody PizzaDto dto`.
- **Validar configuración**: `@NotNull` en propiedades de un `@ConfigurationProperties`.
- **Documentar restricciones de almacenamiento**: en una entidad JPA, dice "esto debería ser así cuando se persista".

Lo que **no es**: un sustituto de las validaciones del factory del dominio. Si pones `@NotBlank` sobre un campo `name` de tu `Pizza` y no validas en `create`, **alguien puede crear `new Pizza()` desde algún flujo y poner `name = ""`** sin que Bean Validation se entere, porque solo se ejecuta cuando algo (un controlador, JPA) lo invoca explícitamente.

> **Pedro, si quieres usarlo**: úsalo en la entidad JPA como **complemento** (3.8 lo cubre), no en el dominio como sustituto.

## Validaciones complejas: cuando una excepción no basta

A veces la validación no es "pasa o falla". A veces queremos **acumular errores** y devolverlos juntos. Por ejemplo, al crear un `User`:

- El nombre está vacío.
- El email está vacío.
- El email no tiene `@`.

Lanzar una excepción al primer fallo significa que el usuario corrige el nombre, vuelve a probar, descubre que falla el email, lo corrige, vuelve a probar, descubre que el formato sigue mal… UX horrible.

Hay dos maneras de tratarlo:

### Opción A — el dominio sigue siendo estricto, la validación de UI es otra capa

```java
// En la capa de aplicación / controlador
List<String> errores = new ArrayList<>();
if (dto.getName() == null || dto.getName().isBlank())
    errores.add("nombre obligatorio");
if (dto.getEmail() == null) errores.add("email obligatorio");
else if (!dto.getEmail().contains("@")) errores.add("email inválido");
if (!errores.isEmpty()) throw new ValidationException(errores);

// El dominio no se entera; recibe datos ya correctos
User u = User.create(dto.getName(), dto.getEmail());
```

El dominio **no acumula errores**, lanza al primero. Quien quiera UX agradable, valida antes.

### Opción B — el dominio expone un método que valida sin construir

Más expresivo, pero ya estás duplicando lógica entre `validar` y `create`.

**Para el curso**, vamos con la opción A: **el dominio falla rápido, la UX se construye en la capa de aplicación**. Es más limpio.

---

## Resumen

- Una **invariante** es una propiedad que **siempre** debe cumplirse mientras el objeto exista.
- Las invariantes viven **en el factory `create` y en los métodos de mutación** (`renombrar`, `actualizarCoste`...) de la entidad de dominio. **Nunca en `restore`**.
- Validar en el controlador es bueno **como UX**, pero no protege el dominio. Otros flujos podrían crear objetos sin pasar por ahí.
- Bean Validation en la entidad JPA es **complemento**, no sustituto.
- **Validación de entrada**, **invariantes de dominio** y **constraints de almacenamiento** son tres cosas distintas que se complementan en capas. No se duplican.
- Cuando una invariante puede vivir en un **value object propio** (como `FechaComentario`), llévala ahí. La cura para *primitive obsession* es **mover las invariantes al tipo más pequeño que pueda contenerlas**.
- Las invariantes que dependen de **información externa** (unicidad, existencia) **no son invariantes de la entidad**, son del caso de uso o del agregado.
- El dominio **falla rápido**: lanza excepción al primer problema. La UX agradable (acumular errores) se construye en la capa de aplicación.
- Cuando una invariante se rompe, lanzamos `IllegalArgumentException` (o una excepción de dominio propia, como `IngredienteInvalidoException`). **Que explote pronto y claro.**
