# 2.4. Identidad de dominio vs identidad de fila

## La pregunta que casi nadie se hace

> **¿Cuándo dos pizzas son "la misma pizza"?**

Parece una pregunta filosófica. No lo es. Es la pregunta más práctica del modelado de dominio, y la respuesta determina:

- Cómo escribís `equals` y `hashCode`.
- Cómo se comportan las pizzas en `Set`, `HashMap`, `Stream.distinct()`.
- Si dos copias del "mismo" objeto se ven como una o como dos.
- Si vuestra capa de persistencia se va a portar bien o va a hacer cosas raras.

Y como casi nadie la responde explícitamente, la mayoría de proyectos viven con un `equals` por defecto (que compara referencias) o con un `equals` autogenerado por el IDE (que compara todos los campos), y ambos son **trampa**.

---

## Identidad por referencia (la del `==` de Java)

El `equals` por defecto de `Object` compara **referencias**:

```java
Pizza a = Pizza.create("Margarita", ingredientes);
Pizza b = Pizza.create("Margarita", ingredientes);
a.equals(b); // false
```

Para Java, `a` y `b` son objetos distintos porque son **dos posiciones distintas en memoria**. (Y además, en este caso son conceptualmente distintos también: cada `create` genera una `PizzaId` nueva. Ya volveremos a esto.)

Esto rompe casi todo lo que hagáis:

```java
Set<Pizza> catalogo = new HashSet<>();
catalogo.add(a);
catalogo.contains(b); // false
catalogo.add(b);     // se añade como si fuera distinta
catalogo.size();      // 2
```

Conclusión: **el `equals` por defecto no sirve** en cuanto pasáis las entidades por colecciones, mapas, comparaciones, etc.

---

## Identidad por todos los atributos (lo que genera el IDE)

El IDE os ofrece "Generate equals and hashCode → Use all fields". Y esto:

```java
@Override
public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof Pizza)) return false;
    Pizza p = (Pizza) o;
    return Objects.equals(id, p.id)
        && Objects.equals(name, p.name)
        && Objects.equals(description, p.description)
        && Objects.equals(url, p.url)
        && Objects.equals(ingredientes, p.ingredientes);
}
```

Lo que parece bueno: dos pizzas idénticas en todos los campos son iguales.

Lo que está roto:

1. **Si modificáis cualquier campo, el `hashCode` cambia.** Si la pizza estaba en un `HashSet`, deja de encontrarse. Bug muy difícil de localizar. **Y como en nuestro modelo los campos son mutables (la pizza se renombra, se le actualiza la imagen…), este problema es seguro, no hipotético.**
2. **Compara `ingredientes`**, que son otra entidad. Si `Ingrediente.equals` no está bien definido, el `equals` de `Pizza` tampoco. Cadenas de comparaciones que pueden llegar a ciclos.
3. **Conceptualmente es absurdo**: si renombro la pizza "Margarita" a "Margherita" para corregir una falta, ¿deja de ser la misma pizza? Por supuesto que no. Su **identidad** no cambió, solo cambió un atributo.

Por eso decimos que el IDE genera código rápido, no código correcto.

---

## Identidad de dominio: la única que importa de verdad

En **DDD**, una **entidad** se define por su **identidad**, no por sus atributos. Una pizza es la misma pizza si tiene **el mismo identificador**, aunque le cambies el nombre, la descripción y los ingredientes.

```java
@Override
public boolean equals(Object o) {
    return o instanceof Pizza other && this.id.equals(other.id);
}

@Override
public int hashCode() {
    return id.hashCode();
}
```

**Solo el id**. Nada más.

Consecuencias:

- Dos pizzas con el mismo id son la misma pizza, aunque sus campos hayan divergido (típicamente porque una está en memoria y la otra se ha modificado).
- Dos pizzas con el mismo nombre pero distinto id son **dos pizzas distintas**. Y eso es correcto: la "Margarita" de hoy y la "Margarita" de la versión 2 del menú no tienen por qué ser la misma cosa.
- El `hashCode` es **estable**: el id es `final` y no cambia durante la vida del objeto. Es la única razón por la que `final` aparece en una entidad: para que **la identidad** sea inmutable.

---

## Identidad de fila (la del `id` de la base de datos)

Cuando llegamos a JPA, aparece **otro tipo de identidad**: la **clave primaria** de la tabla. Y aquí es donde la mayoría de proyectos se confunden.

La tentación es:

```java
public class Pizza {
    @Id
    @GeneratedValue
    private Long id; // generado por la BD
    // ...
}
```

Y usar ese `id` como identidad de dominio.

**Problema gordo**: ese `id` **no existe hasta que la entidad se persiste**. Hasta que llamáis `em.persist(pizza)` y la BD asigna el id, **el campo es `null`**.

Consecuencias:

- Antes de persistir: `pizza.getId()` es `null`. `equals` da igual a otros `null`. `hashCode` es 0 para todas. **Todas las pizzas no persistidas se consideran "iguales" entre sí**. Si las metéis en un `Set` antes de persistir, se queda solo una.
- Después de persistir: el `id` se asigna y el `hashCode` cambia. Si la pizza estaba en una colección basada en hash, **deja de encontrarse**.

Esto es el clásico "JPA equals/hashCode bug", y hay miles de hilos en Stack Overflow al respecto. La causa raíz es **mezclar identidad de dominio con identidad de fila**.

---

## La solución: identidad de dominio independiente de la BD

La identidad la **decide el dominio**, no la base de datos:

```java
public class Pizza {
    private final PizzaId id; // value object, asignado en create()
    // ...

    public static Pizza create(String name, List<Ingrediente> ingredientes) {
        // ...validaciones...
        return new Pizza(PizzaId.nuevo(), name, ingredientes, ...);
    }

    @Override public boolean equals(Object o) {
        return o instanceof Pizza other && this.id.equals(other.id);
    }
    @Override public int hashCode() { return id.hashCode(); }
}
```

`PizzaId` es un **value object** que envuelve un identificador. Puede ser:

- Un `UUID` generado en el dominio: `PizzaId.nuevo()` → `UUID.randomUUID()`.
- Un valor proporcionado desde fuera (un código de catálogo, por ejemplo).
- Cualquier otra cosa que el negocio decida.

**Lo importante**: el id **existe desde el momento de la construcción** (`create` lo genera; `restore` lo recibe ya hecho). No depende de la BD. No es `null`. No cambia.

Cuando lleguemos a JPA, en la entidad de persistencia traduciremos:

- O bien usamos el mismo `UUID` como clave primaria (`@Id` sobre el `UUID`).
- O bien tenemos en la entidad JPA dos columnas: `id` (técnico, autogenerado) y `domain_id` (el `UUID` del dominio). Y el `equals` del dominio sigue usando `domain_id`.

Cualquiera de las dos funciona. Lo crítico es que el dominio **nunca dependa del id autogenerado de la BD**.

---

## ¿Por qué `PizzaId` y no `UUID` directamente?

Tentación: `private UUID id;`. Funciona. ¿Por qué meter una clase envoltorio?

```java
// Sin value object
public static Pizza create(UUID id, ...) { ... }

// Con value object
public static Pizza create(PizzaId id, ...) { ... }
```

Razones (las mismas que vimos en 2.2 con `Money`, aplicadas aquí):

1. **Legibilidad**. `Pizza.create(pizzaId, ...)` deja claro qué tipo de id es. `Pizza.create(uuid, ...)` no.
2. **Tipo seguro**. Si tenéis `PizzaId` y `IngredienteId` como tipos distintos, **no podéis pasar uno donde se espera el otro**. Con `UUID` para todo, sí podéis, y el compilador no os avisa.
3. **Encapsulación**. Si mañana decidís cambiar de `UUID` a `String` o a `Long`, **un solo sitio** se modifica.
4. **Comportamiento**. `PizzaId.nuevo()` es un método estático bonito. `UUID.randomUUID()` es ruido.

Es la cura de la *primitive obsession* aplicada a los identificadores: **encapsular tipos primitivos en value objects de dominio**.

Y como los ids son value objects (inmutables, sin identidad propia, equals/hashCode por valor), el `record` los expresa perfectamente:

```java
public record PizzaId(UUID value) {
    public PizzaId {
        if (value == null) throw new IllegalArgumentException("id obligatorio");
    }
    public static PizzaId nuevo() { return new PizzaId(UUID.randomUUID()); }
    public static PizzaId de(UUID v) { return new PizzaId(v); }
    public static PizzaId de(String v) { return new PizzaId(UUID.fromString(v)); }
}
```

Pequeño, simple, blindado. El `record` nos regala equals/hashCode por valor (que es exactamente lo que necesitamos en un id), constructor que valida (compact constructor), accesores y `toString`. Lo mismo se aplica a `IngredienteId`, `CommentId`, `UserId`.

---

## El `toString` y los ciclos infinitos

Aviso de campo de minas:

```java
@Override
public String toString() {
    return "Pizza{id=" + id + ", ingredientes=" + ingredientes + "}";
}
```

Si `Ingrediente.toString()` incluye sus pizzas (en una bidireccional), tenéis **recursión infinita**. `StackOverflowError` al primer `log.info(pizza)`. Es un clásico.

Reglas:

- En `toString` de una entidad, **incluid solo el id (y opcionalmente atributos simples como nombre)**. Nunca colecciones de otras entidades.
- `equals` y `hashCode` también: solo el id. Ya lo dijimos arriba; ahora se entiende también desde otro ángulo.

---

## Resumen de los tres tipos de identidad que vais a manejar

| Identidad | Quién la decide | Cuándo se asigna | Usar en `equals`? |
|---|---|---|---|
| **De referencia** (`==`) | La JVM | Al crear el objeto | Solo si querés "el mismo objeto en memoria". Nunca para entidades. |
| **De dominio** (`PizzaId`) | El dominio | En `create()` o recibida en `restore()` | **Sí**. Es la única correcta para entidades de dominio. |
| **De fila** (`@GeneratedValue Long`) | La BD | Al persistir | **Nunca** para `equals` del dominio. Solo internamente en la entidad JPA. |

Las tres existen. Las tres tienen su rol. **Mezclarlas es donde nacen los bugs**.

---

## Aplicación al modelo del curso

| Entidad | Tipo de id | Estrategia |
|---|---|---|
| `Pizza` | `PizzaId` (record sobre UUID) | Generado en el dominio en `create()`. |
| `Ingrediente` | `IngredienteId` (record sobre UUID) | Generado en el dominio en `create()`. |
| `Comment` | `CommentId` (record sobre UUID) | Generado en el dominio en `create()`. |
| `User` | `UserId` (record sobre UUID) | Generado en el dominio en `create()`. |

Todos los `equals`/`hashCode` usan **solo el id**. Punto.

En el bloque 3, cuando hagamos las entidades JPA, tendremos que decidir si el `UUID` es la PK de la tabla o si tenemos un `Long` autogenerado adicional. Esa es una decisión de **persistencia**, no de dominio. La separación es exactamente lo que protege que no se contaminen.

---

## Resumen

- **Identidad de dominio ≠ identidad de fila ≠ identidad de referencia.** Tres conceptos, tres usos.
- En entidades de dominio, `equals` y `hashCode` se basan **solo en el id de dominio**.
- El id de dominio se asigna **en `create()`** (que llama a `PizzaId.nuevo()`) o se recibe **en `restore()`** desde persistencia. Nunca lo asigna la BD.
- El `id` es **el único campo `final`** de una entidad de dominio. Es la única razón por la que `final` aparece: para que **la identidad** sea inmutable mientras el resto puede cambiar.
- El antipatrón clásico es usar `@GeneratedValue Long` como identidad de dominio. Antes de persistir es `null`; después de persistir cambia. Rompe colecciones, rompe `Set`, rompe todo.
- Encapsulad los ids en **value objects implementados como `record`** (`PizzaId`, `UserId`...). El `UUID` crudo no transmite intención, permite errores que el compilador no detecta y obliga a escribir equals/hashCode a mano.
- En `toString`, incluid el id y poco más. Nunca colecciones de otras entidades. Recursión infinita garantizada.
- Esta decisión, tomada el primer día, **os ahorra horas de debug** cuando lleguemos a JPA y al ciclo de vida del `EntityManager`.
